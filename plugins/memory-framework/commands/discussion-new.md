# /discussion-new — Заведение обсуждения

> Длинное обсуждение (≥3 раундов уточнений по одной теме) лучше вести в
> структурированной форме: WIP-файл как живая карта + design.md как
> накапливаемый финал. Эта команда создаёт каркас.

## Аргументы

- `topic-slug` — kebab-case имя обсуждения (например, `auth-refactor`,
  `db-migration-strategy`). Станет именем папки.
- `короткое описание темы` — для `description` в frontmatter и для
  строки в `INDEX.md`.

Если аргументы не переданы — спросить у пользователя.

## Алгоритм

### 1. Определи путь к памяти — развилка «проект / блокнот»

Два режима. **Проектный**: у места работы есть своя папка обсуждений —
обсуждение живёт в ней (едет с git, видно человеку). **Блокнот**: иначе —
fallback в auto-memory проекта сессии (машинно-локальный слой).

```bash
# Проектный режим: у корня места есть своя папка обсуждений?
PROJ_ROOT=$(git rev-parse --show-toplevel 2>/dev/null || pwd)
MODE=notebook
if [ "$PROJ_ROOT" != "$HOME" ]; then
  for CAND in "$PROJ_ROOT/.obsidian-docs/discussions" "$PROJ_ROOT/discussions"; do
    [ -d "$CAND" ] && DISC_DIR="$CAND" && MODE=project && break
  done
fi
if [ "$MODE" = notebook ]; then
  # Блокнот якорится на проект СЕССИИ, а не на pwd (pwd мог уехать за сессию).
  PROJECT_DIR=$(dirname "$(find ~/.claude/projects -maxdepth 2 -name "$CLAUDE_CODE_SESSION_ID.jsonl" 2>/dev/null | head -1)")
  [ -d "$PROJECT_DIR" ] || PROJECT_DIR=~/.claude/projects/$(pwd | sed 's|/|-|g')
  DISC_DIR="$PROJECT_DIR/memory/discussions"
fi
```

Если пользователь явно просит завести **проектные** обсуждения там, где
папки ещё нет (первое обсуждение в месте) — создай `"$PROJ_ROOT/discussions"`
и работай в проектном режиме. По умолчанию без явной просьбы — блокнот.

Дальше во всех шагах используй `$DISC_DIR`, а не литеральные пути.

### 2. Проверь что обсуждение не существует

```bash
test -d "$DISC_DIR/<topic-slug>" && echo "Уже существует" && exit
test -d "$DISC_DIR/_archive/<topic-slug>-*" && echo "Есть в архиве" && exit
```

### 2.5. Проверь наличие handoff

```bash
HANDOFF=~/claude-dotfiles/handoffs/<topic-slug>.md
```

Если файл существует — это обсуждение, начатое на другой машине.

1. Прочитай handoff целиком.
2. Используй его содержимое для заполнения WIP.md на шаге 4:
   - `## Контекст` ← из handoff `## Контекст`
   - `## Что РЕШЕНО` ← из handoff `## Ключевые решения`
   - `## Что ОТКРЫТО` ← из handoff `## Открытые вопросы`
   - `## Применённые изменения` ← из handoff `## Применённые изменения`
3. Добавь в WIP блок:
   ```
   > Восстановлено из handoff (<дата экспорта из frontmatter>).
   > Ход рассуждений предыдущей сессии — см. ~/claude-dotfiles/handoffs/<topic-slug>.md
   ```
4. Скажи пользователю: «Найден handoff от <дата>, обсуждение восстановлено.»
5. Описание темы для INDEX.md возьми из handoff `description`.

Если handoff **не найден** — продолжай обычный flow (описание берётся из аргумента).

### 3. Получи UUID текущей сессии

```bash
SESSION_ID=$CLAUDE_CODE_SESSION_ID
```

### 4. Создай папку и WIP.md

`$DISC_DIR/<topic-slug>/WIP.md`:

```markdown
---
name: <Topic> WIP
description: <короткое описание темы — что обсуждаем и зачем>
type: project
originSessionId: <SESSION_ID>
---
# <Topic> — WIP

> **WIP = Work In Progress.** Рабочий файл, обновляется по ходу обсуждения.
> Финальные решения переносятся в `./design.md`.
> Если был автокомпакт — начинай чтение отсюда.

## Контекст

<1-2 абзаца: почему эта тема возникла, что важно знать для возобновления>

## Что РЕШЕНО

(пусто на старте — заполняется по ходу)

## Что ОТКРЫТО

(перечисляй вопросы, которые нужно закрыть для финала)

## Применённые изменения

(что физически уже сделано в файлах/конфигах в рамках обсуждения)

## Связанные файлы

- `../INDEX.md` — индекс активных обсуждений
- `../../MEMORY.md` — глобальный индекс памяти
```

### 4.5. Проектный режим: самонастройка якоря в CLAUDE.md

**Только при `MODE=project`.** Смысл: WIP активного обсуждения проходит
критерий `@file`-якоря (нужен в каждой сессии места + его игнор ведёт к
потерям после автокомпакта), поэтому якорь ставится автоматически — человек
не должен знать про эту механику.

Путь якоря — относительный от `$PROJ_ROOT`:
`ANCHOR=@<относительный-путь-к-DISC_DIR>/<topic-slug>/WIP.md`
(например `@discussions/auth/WIP.md` или `@.obsidian-docs/discussions/auth/WIP.md`).

- **`$PROJ_ROOT/CLAUDE.md` нет** → создай:

  ```markdown
  # <имя папки места>

  Рабочая область. Активные обсуждения живут в `<DISC_DIR относительный>/`
  и подключены якорями ниже — любую сессию начинать с их актуального
  состояния.

  ## Правила работы
  - Рабочий файл обсуждения — `WIP.md` в папке обсуждения: живая карта
    (решено / открыто / применённые изменения). Обновлять по ходу ключевых
    реплик, не «в конце».
  - После автокомпакта или новой сессии — сначала перечитать WIP активного
    обсуждения, потом продолжать.
  - Материалы и сырьё — отдельными файлами в `notes/` внутри папки
    обсуждения; в WIP — только решения и ссылки на файлы.

  ## Активные обсуждения (якоря)
  <ANCHOR>
  ```

- **`$PROJ_ROOT/CLAUDE.md` есть** → добавь строку `<ANCHOR>` в секцию
  `## Активные обсуждения (якоря)` (создай секцию в конце файла, если её
  нет). Ничего другого в чужом CLAUDE.md не трогай.

Правило стерильности: якорь — это инструкция (указатель), в CLAUDE.md
легален. Содержание обсуждения в CLAUDE.md не пишется никогда.

### 5. Обнови INDEX.md

Если `$DISC_DIR/INDEX.md` нет (первое обсуждение места) — создай с
секциями `## Активные` и `## Архив` (без frontmatter — это индекс).

Добавь строку в `## Активные` секцию `discussions/INDEX.md`:

```markdown
- **[<topic-slug>](<topic-slug>/)** — <короткое описание темы>.
  Статус: active. Точка входа: [WIP.md](<topic-slug>/WIP.md).
```

### 6. Отчёт пользователю

```
Обсуждение заведено: discussions/<topic-slug>/

- WIP: $DISC_DIR/<topic-slug>/WIP.md
- Статус: active
- INDEX.md обновлён

Дальше: пиши свободным текстом, я буду обновлять WIP по ключевым решениям.
Закрытие — командой /discussion-close <topic-slug>.
```

## Правило ведения обсуждения: сверка перед «двигаемся дальше»

Это обсуждение ведётся в WIP. Когда по ходу диалога ты приходишь к выводу,
что **очередной узел закрыт и можно двигаться дальше** (типа «ок, с этим
разобрались», «вопрос решён», «переходим к следующему») — **сначала сделай
лёгкую сверку, потом озвучивай вывод.** Не объявляй узел закрытым «на ощупь».

Лёгкая сверка (дешёвая, без похода в файлы проекта):

1. **Полнота WIP.** Перечитай `## Что ОТКРЫТО`. Пункт, который ты собрался
   объявить закрытым, — он реально закрыт принятым решением, или просто
   «обсудили и забыли»? Нет ли решений, которые противоречат друг другу?
2. **Незаписанные решения.** Есть ли в нашем разговоре решения/ответы,
   которые приняты, но ещё не попали в `## Что РЕШЕНО`? Если да —
   **сразу обнови WIP** (не спрашивая разрешения), потом продолжай.

Если сверка чистая — фиксируй решение в WIP и озвучивай «двигаемся дальше».

Если сверка нашла дыру (висящий узел, противоречие, незакрытый кусок) —
**не объявляй узел закрытым.** Покажи короткий список найденного и верни
в обсуждение. Дообсуждать или оставить как есть — **решает пользователь**,
не ты.

> Полная сверка (с походом в реальное состояние проекта и проверкой
> покрытия исходной цели) делается при закрытии — см. `/discussion-close`.

## Правило вывода

После создания — **не печатай содержимое WIP.md в чат**. Только статус
и пути.

## Примечание про режимы

Развилка «проект / блокнот» реализована 2026-07-13 (решение обсуждения
memory-organization): есть у места своя папка обсуждений → ведём там
(+ автоякорь в CLAUDE.md места), иначе блокнот (auto-memory проекта сессии,
якорится через `CLAUDE_CODE_SESSION_ID`, не через `pwd`). Та же развилка —
в `/discussion-close` и `/discussion-handoff`; менять только синхронно.
