# /discussion-close — Закрытие обсуждения

> Запускается, когда обсуждение по теме `<topic-slug>` достигло финала
> и решения зафиксированы в WIP. Команда переписывает `design.md`,
> архивирует папку, обновляет `INDEX.md`.

## Аргументы

Вход — тема в свободной форме или имя папки обсуждения. Сопоставь её с
активными обсуждениями из `INDEX.md` (по именам папок и описаниям) и
определи `<topic-slug>` сам. Если аргумент не передан или совпадение
неоднозначно — покажи список активных обсуждений человеческим языком
(описание + имя папки) и дай выбрать. Технические термины («slug»,
«kebab-case») в диалоге с пользователем не употреблять.

## Алгоритм

### 1. Валидация

Обсуждение ищется в папке обсуждений рабочего места — та же логика,
что в `/discussion-new`.

```bash
PROJ_ROOT=$(git rev-parse --show-toplevel 2>/dev/null || pwd)
DISC=""
for CAND in "$PROJ_ROOT/.obsidian-docs/discussions" "$PROJ_ROOT/discussions"; do
  [ -d "$CAND/<topic-slug>" ] && DISC="$CAND/<topic-slug>" && break
done
[ -n "$DISC" ] || { echo "Не найдено активное обсуждение <topic-slug> в $PROJ_ROOT"; exit; }
DISC_DIR=$(dirname "$DISC")
test -f "$DISC/WIP.md" || { echo "WIP.md отсутствует"; exit; }
```

Если не найдено, а пользователь говорит, что обсуждение точно есть —
возможно, оно заведено старой версией команды в машинно-локальном слое.
Поищи `~/.claude/projects/*/memory/discussions/<topic-slug>` (если такой
путь на машине существует), **покажи найденный полный путь и получи
подтверждение пользователя**, что это то самое обсуждение. Предложи
сначала перенести папку в `discussions/` рабочего места (чтобы финал
остался в проекте), затем закрывай.

### 2. Прочитай WIP.md

Внимательно. WIP содержит:
- Контекст
- Что РЕШЕНО (с нумерацией решений)
- Что ОТКРЫТО (то, что НЕ закрыто намеренно — оставить как есть)
- Применённые изменения

### 2.5. Полная сверка перед закрытием — ОБЯЗАТЕЛЬНЫЙ шаг

**Нельзя пропускать.** Прежде чем фиксировать финал, проверь, что WIP
действительно полон и соответствует реальности. Это та самая ревизия всех
узлов, ради которой обсуждение и закрывается осознанно, а не «на ощупь».

Четыре измерения сверки:

1. **WIP ↔ состояние проекта.** Возьми раздел `## Применённые изменения`
   и пройдись по реальным файлам/конфигам/коду. Совпадает ли заявленное
   с фактом? Нет ли решений из `## Что РЕШЕНО`, которые подразумевают
   изменения, но в проекте их нет (или сделано иначе)? Читай реальные
   файлы — не верь WIP на слово.
2. **Внутренняя полнота WIP.** Все ли пункты `## Что ОТКРЫТО` либо закрыты
   решением, либо осознанно отложены? Нет ли висящих узлов и взаимных
   противоречий между решениями?
3. **Незаписанные решения из беседы.** Есть ли в этой сессии решения,
   которые приняты, но не попали в WIP? Если да — **сначала обнови WIP**,
   потом продолжай.
4. **Покрытие исходной цели.** Перечитай `## Контекст` (зачем тема
   возникла, что хотели решить). Все ли аспекты исходной цели закрыты
   решениями? Нет ли куска, который молча обошли — его нет ни в РЕШЕНО,
   ни в ОТКРЫТО? Это самый частый источник дыр.

**Если сверка чистая** — скажи «Сверка перед закрытием чистая,
расхождений нет» и переходи к шагу 3.

**Если сверка нашла дыры** (расхождение с проектом, висящий узел,
непокрытый аспект цели, противоречие) — **не закрывай обсуждение.**
Покажи пользователю короткий структурированный список найденного:
что именно, в каком измерении, чем грозит. Дальше — **решает
пользователь**: дообсудить и закрыть дыры, либо закрыть как есть,
зафиксировав их в `## Что НЕ закрыто (намеренно)` финального design.
Не закрывай молча и не дорешай за пользователя.

### 3. Перепиши design.md с нуля

Существующий `design.md` (если есть) **полностью заменяется** на новый,
основанный на актуальном WIP. Структура:

```markdown
---
name: <Topic> design (final)
description: <обновлённое одно-предложение что решено>
type: project
originSessionId: <UUID текущей сессии>
---
# <Topic> — финальный дизайн

> Финал обсуждения, закрытого <YYYY-MM-DD>. Источник правды по теме.
> Если правка — править дизайн, а не разводить новое обсуждение,
> пока не появится принципиально новый угол.

## Цель

<1 абзац: что решали и зачем>

## <Разделы по существу решений>

<Развернуть пронумерованные решения из WIP в связный текст по
тематическим разделам. НЕ копируй WIP буквально — делай связное
изложение финала.>

## Что НЕ закрыто (намеренно)

<Перенести из WIP — то что отложено и почему>

## Применённые изменения

<Список физических изменений в файлах/конфигах>

## Связанные обсуждения

<Ссылки на родственные обсуждения (если есть)>
```

### 4. Архивация

```bash
TODAY=$(date +%Y-%m-%d)
ARCHIVE_DIR="$DISC_DIR/_archive/<topic-slug>-$TODAY"
mkdir -p "$DISC_DIR/_archive"
mv "$DISC_DIR/<topic-slug>" "$ARCHIVE_DIR"
```

### 4.5. Сними якорь из CLAUDE.md

Обсуждение закрыто — его WIP больше не проходит критерий якоря. Найди в `$PROJ_ROOT/CLAUDE.md` строку-якорь,
содержащую `<topic-slug>/WIP.md` (обычно в секции
`## Активные обсуждения (якоря)`), и удали её. Если секция опустела —
удали и секцию. Ничего другого в CLAUDE.md не трогай. Дорожка к финалу
остаётся через INDEX.md (шаг 5) — отдельный текстовый указатель в
CLAUDE.md не нужен.

### 5. Обнови INDEX.md

В `discussions/INDEX.md`:
- Удали строку про `<topic-slug>` из секции `## Активные`.
- Добавь строку в секцию `## Архив`:

```markdown
- **[<topic-slug>-<YYYY-MM-DD>](_archive/<topic-slug>-<YYYY-MM-DD>/)**
  — <обновлённое описание из design.description>. Закрыто <YYYY-MM-DD>.
  Финал — [design.md](_archive/<topic-slug>-<YYYY-MM-DD>/design.md).
```

### 6. (Опционально) Постоянное знание места

Если из обсуждения выросло **постоянное знание** (правило / факт,
полезный за пределами закрытой темы) — спроси пользователя:
> «Из обсуждения вышли решения, которые стоит сохранить как постоянное
> знание: <перечисление>. Создать файлы?»

При согласии — создай файлы рядом с папкой обсуждений (в корне рабочего
места, в `notes/` или куда укажет пользователь), один файл = один факт,
каждый с шапкой:

```yaml
---
name: <короткое-имя-kebab-case>
description: <одна конкретная строка-крючок — по ней решают, грузить ли файл>
type: <user | feedback | project | reference>
originSessionId: <UUID текущей сессии>
---
```

### 7. Отчёт пользователю

```
Обсуждение <topic-slug> закрыто:

- design.md переписан → _archive/<topic-slug>-<YYYY-MM-DD>/design.md
- WIP.md перемещён вместе с папкой
- INDEX.md обновлён (Активные → Архив)
- Постоянная память: <создано N файлов | не выносилось>

Дальше: возобновление возможно через /discussion-new <new-topic>,
если появится принципиально новый угол.
```

## Правило

Не печатай содержимое design.md в чат после создания. Только статус
и пути.

## Если пользователь спросит, что это за команда

Первый ответ — каноническим текстом (дословно или очень близко):

> Это подведение итога по теме. Я сверяю карту обсуждения с тем, что
> реально сделано, собираю итоговый документ с принятыми решениями и
> переношу тему из активных в архив. Итог остаётся в проекте — к нему
> всегда можно вернуться.

Дальше — своими словами по тому же правилу: без технических терминов
(WIP, frontmatter, slug, якорь, CLAUDE.md) — говори «живая карта»,
«папка темы», «итоговый документ». Технические подробности — только
если спросят прямо.
